# 트러블슈팅 기록

이 문서는 React Webpack Starter 프로젝트를 구성하면서 실제로 마주친 문제와 해결 과정을 정리한 기록입니다.

`webpack-decisions.md`가 설정 선택의 이유를 설명하는 문서라면, 이 문서는 구현 과정에서 어떤 문제가 발생했고 어떤 방식으로 해결하거나 검증했는지에 초점을 둡니다.

---

## 1. SPA 라우트에 직접 접근했을 때 404가 발생한 문제

### 문제

React Router로 `/analytics`, `/customers`, `/settings` 같은 라우트를 구성한 뒤, 브라우저 주소창에 경로를 직접 입력하거나 새로고침했을 때 404 에러가 발생했다.

### 원인

SPA는 클라이언트가 라우팅을 처리하지만, 직접 URL 접근 시에는 먼저 개발 서버가 해당 경로에 대응하는 정적 파일을 찾으려 한다. 이때 서버에 실제 파일이 없으면 404가 발생한다.

### 해결

개발 서버에 `historyApiFallback`을 적용해 모든 요청이 `index.html`로 fallback 되도록 설정했다.

### 결과

적용 후에는 직접 URL 접근이나 새로고침 시에도 404 없이 정상적으로 클라이언트 라우터가 경로를 해석할 수 있었다.

### 배운 점

React Router 같은 SPA 라우팅은 프론트엔드 코드만으로 완성되지 않는다.  
개발 서버에서는 fallback 설정이 필요하고 배포 서버에서는 rewrite 설정이 필요하다.

---

## 2. `React.lazy`를 적용했는데 실제로 번들이 분리된 것인지 확신하기 어려웠던 문제

### 문제

페이지 컴포넌트에 `React.lazy(() => import(...))`를 적용했지만, 코드를 분리했다는 사실만으로 실제 번들이 원하는 방식으로 나뉘었는지 확신하기 어려웠다.

### 원인

`React.lazy`는 비동기 import를 적용하는 React 문법이지만, 설정만 추가했다고 해서 실제 빌드 결과와 브라우저 요청 흐름이 기대한 대로 동작하는지는 별도로 검증해야 한다.

### 해결

`webpack-bundle-analyzer`로 production 번들 구조를 확인하고, 브라우저 Network 탭에서 초기 진입 시점과 특정 라우트 이동 시점의 요청 파일 변화를 비교했다.

### 결과

초기 진입 시에는 필요한 엔트리 번들만 로드되고, 특정 라우트에 진입할 때 해당 페이지 청크가 추가 요청되는 흐름을 확인할 수 있었다.

### 배운 점

코드 스플리팅은 설정 자체보다 실제 결과 검증이 중요하다.  
Analyzer는 정적인 번들 구조를 확인하는 데 유용했고, Network 탭은 실제 요청 흐름을 확인하는 데 유용했다.

---

## 3. 라우트 단위 분리만으로는 공통 코드 구조를 이해하기 어려웠던 문제

### 문제

처음에는 `React.lazy`만 적용하면 충분히 번들이 나뉜다고 생각했지만,  
공통 의존성이 어떤 식으로 묶이는지까지는 명확히 보이지 않았다.

### 원인

페이지 단위 lazy loading은 라우트별 청크 분리에는 도움이 되지만, 공통 라이브러리나 공유 코드가 어떤 구조로 분리되는지는 별도의 최적화 전략이 필요하다.

### 해결

production 설정에 `splitChunks`와 `runtimeChunk: "single"`을 적용하고, Bundle Analyzer로 초기 엔트리 번들 / 공통 청크 / 비동기 청크 구조를 함께 확인했다.

### 결과

적용 후에는 라우트별 비동기 청크 외에 공통 청크가 별도로 분리된 구조를 확인할 수 있었다.
다만 현재는 `cacheGroups`를 직접 세밀하게 설계했다기보다, 기본 분리 전략을 통해 공통 의존성이 나뉘는 흐름을 확인하는 데 초점을 두었다.

### 배운 점

`React.lazy`와 `splitChunks`는 역할이 다르다.  
하나는 “언제 로드할지”에 가깝고, 다른 하나는 “어떻게 묶을지”에 더 가깝다는 점을 구분해서 볼 필요가 있었다.

## 4. 운영 빌드에서 CSS가 JS 번들과 함께 묶이는 구조를 분리하고 싶었던 문제

### 문제

개발 환경에서는 `style-loader`로 스타일을 빠르게 주입할 수 있었지만, 운영 빌드에서도 같은 방식을 유지하면 스타일 코드가 JS 번들 안에 함께 포함되는 구조가 된다.

### 원인

개발 환경에서는 빠른 반영이 중요하지만, 운영 환경에서는 스타일과 스크립트를 분리해 관리하는 편이 더 적절하다.  
특히 초기 JS 번들에 스타일 코드가 함께 포함되면 번들 구조를 해석하기도 어렵고, 정적 리소스 분리 관점에서도 아쉬움이 남았다.

### 해결

production 환경에서 `MiniCssExtractPlugin`을 적용해 CSS를 별도 파일로 추출했다.

### 결과

운영 빌드 결과물 기준으로 스타일 코드가 JS 번들 내부에 포함되는 대신 `dist`에 별도 CSS 파일이 생성되는 것을 확인했다.  
또한 entrypoint size warning 완화에도 도움이 됐다.

### 배운 점

개발 환경과 운영 환경은 CSS 처리 방식도 달라야 한다.  
개발에서는 빠른 주입이 중요하고, 운영에서는 분리된 정적 자산 구조가 더 적절하다는 점을 확인할 수 있었다.

---

## 정리

이 프로젝트의 트러블슈팅은 대부분 “Webpack 설정을 추가했다”에서 끝나는 문제가 아니라,  
설정이 실제 브라우저 동작과 빌드 결과에 어떤 영향을 주는지 직접 확인하고 해석하는 과정에서 발생했다.

특히 다음을 중요하게 확인했다.

- SPA 라우팅은 서버 fallback 설정까지 포함해야 한다.
- `React.lazy`는 적용 자체보다 실제 요청 흐름 검증이 중요하다.
- 공통 청크 분리는 lazy loading과 별개로 이해해야 한다.
- CSS 처리 방식도 개발 환경과 운영 환경을 구분해 설계해야 한다.
