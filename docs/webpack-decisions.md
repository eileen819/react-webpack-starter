# Webpack 설정 결정 기록

이 문서는 React Webpack Starter 프로젝트에서 Webpack 설정을 어떤 기준으로 나누고, 왜 해당 옵션을 선택했는지 기록한 문서입니다.

README에서는 전체 구조와 결과를 요약하고, 이 문서에서는 설정 항목별 의사결정 배경과 선택 이유, 대안, 적용 결과를 정리합니다.

---

## 1. 설정 파일을 common / dev / prod로 분리한 이유

### 결정

Webpack 설정을 `webpack.common.js`, `webpack.dev.js`, `webpack.prod.js`로 분리했다.

### 배경

개발 환경과 운영 환경은 목적이 다르다.  
개발 환경에서는 빠른 피드백, HMR, 디버깅 편의성이 중요하고, 운영 환경에서는 번들 최적화, 공통 청크 분리, CSS 추출, `output` 정리 같은 요소가 더 중요하다.

### 선택한 이유

공통 설정과 환경별 설정을 분리하면 각 파일의 역할이 명확해지고, 어떤 옵션이 어느 환경에만 필요한지 쉽게 파악할 수 있다. 특히 이 프로젝트는 Webpack을 직접 구성하고 동작 원리를 검증하는 것이 목적이었기 때문에, 설정 책임을 분리해 구조를 명확하게 드러내는 것이 중요했다.

### 고민한 대안

- 하나의 `webpack.config.js` 파일에서 `mode` 기준 분기 처리
- 작은 프로젝트이므로 단일 파일로도 관리 가능

### 결과 / 영향

환경별 옵션의 역할이 더 분명해졌고, dev/prod 설정이 섞이지 않아 유지보수와 학습 측면에서 모두 유리했다.

---

## 2. 개발 환경에서 webpack-dev-server를 사용한 이유

### 결정

개발 환경에서는 `webpack-dev-server`를 사용하고 HMR을 적용했다.

### 배경

코드를 수정할 때마다 전체 빌드를 다시 수행하고 브라우저를 새로고침하는 방식은 피드백 루프가 길어지고 개발 경험이 나빠진다.

### 선택한 이유

`webpack-dev-server`는 메모리 상에서 번들을 제공하고, 변경 사항을 빠르게 반영할 수 있어 개발 중 반복 작업 비용을 줄여준다. 또한 HMR을 함께 적용하면 전체 페이지를 새로고침하지 않고도 변경 내용을 빠르게 확인할 수 있다.

### 고민한 대안

- `webpack --watch` 기반 감시 모드
- 수동 새로고침 중심 개발

### 결과 / 영향

스타일 수정, 라우트 전환 확인, 컴포넌트 구조 점검 등 개발 반복 작업이 더 빠르게 이루어졌다.

---

## 3. SPA 직접 진입 대응을 위해 historyApiFallback을 적용한 이유

### 결정

개발 서버에 `historyApiFallback`을 적용했다.

### 배경

React Router 기반 SPA는 클라이언트가 라우팅을 처리한다.  
따라서 브라우저에서 `/analytics`, `/settings` 같은 경로로 직접 접근하거나 새로고침하면 서버는 해당 경로에 대응하는 실제 파일을 찾으려 하고, 정적 파일이 없으면 404를 반환할 수 있다.

실제로 `historyApiFallback` 적용 전에는 URL로 직접 접근했을 때 404 에러를 확인했다.

### 선택한 이유

모든 요청을 `index.html`로 fallback 시키면, 이후 경로 해석은 클라이언트 라우터가 담당할 수 있다.  
이 설정은 SPA에서 직접 진입과 새로고침을 정상적으로 처리하기 위한 기본 조건이라고 판단했다.

### 결과 / 영향

`historyApiFallback` 적용 후에는 직접 URL 접근과 새로고침 시에도 404 없이 정상적으로 라우팅 흐름을 확인할 수 있었다.  
또한 실제 배포 환경에서도 동일한 동작을 위해 별도 rewrite 설정이 필요하다는 점을 함께 정리할 수 있었다.

---

## 4. 페이지 단위 코드 스플리팅에 React.lazy + import()를 사용한 이유

### 결정

페이지 컴포넌트는 `React.lazy(() => import(...))`로 불러오도록 구성했다.

### 배경

초기 번들에 모든 페이지 코드를 포함하면 첫 진입 시 필요하지 않은 화면 코드까지 함께 로드된다.  
이 프로젝트는 라우트 단위 코드 스플리팅이 실제로 어떻게 동작하는지 검증하는 것이 주요 목표 중 하나였다.

### 선택한 이유

`React.lazy(() => import(...))`를 사용하면 페이지 단위로 비동기 청크를 나눌 수 있고, 해당 라우트에 진입하는 시점에만 필요한 코드를 불러오도록 구성할 수 있다.  
학습 목적상 가장 직관적이면서도 React Router와 연결해 검증하기 좋은 방식이었다.

### 고민한 대안

- 모든 페이지를 정적 import
- 더 세밀한 컴포넌트 단위 lazy loading

### 결과 / 영향

초기 번들에 모든 페이지가 포함되지 않도록 분리할 수 있었고, 브라우저 Network 탭에서 특정 라우트 진입 시 해당 청크가 추가 요청되는 흐름을 확인할 수 있었다.

---

## 5. 운영 환경에서 splitChunks를 적용한 이유

### 결정

production 환경에 `splitChunks`를 적용했다.

### 배경

페이지 단위 lazy loading만 적용하면 라우트별 청크는 분리할 수 있지만, 공통 의존성까지 어떤 식으로 묶일지는 별도 최적화가 필요하다고 판단했다.

### 선택한 이유

공통으로 사용되는 코드와 의존성을 별도 청크로 분리해 번들 구조를 더 명확하게 확인하고 싶었다.  
이 프로젝트는 코드 스플리팅이 실제 번들 결과에 어떻게 반영되는지 검증하는 것이 목적이었기 때문에, 라우트별 청크 분리 외에 공통 청크 분리도 함께 적용했다.

### 결과 / 영향

`splitChunks` 적용 후 Bundle Analyzer에서 초기 엔트리 번들 외에 공통 청크가 별도로 분리된 구조를 확인할 수 있었다.  
다만 현재는 `cacheGroups`를 직접 세밀하게 설계한 것은 아니며, 기본 분리 전략을 바탕으로 공통 의존성이 별도 청크로 나뉘는 흐름을 확인하는 데 초점을 두었다.

---

## 6. runtimeChunk: "single"을 적용한 이유

### 결정

`runtimeChunk: "single"`을 적용했다.

### 배경

Webpack runtime 코드는 각 청크 간 로딩 관계를 관리하는 역할을 한다.  
이 코드가 엔트리마다 분산되면 번들 구조를 이해하기 어렵고, 캐싱 관점에서도 변경 영향이 커질 수 있다.

### 선택한 이유

런타임 코드를 별도 청크로 분리하면 번들의 역할 구분이 더 명확해지고, 엔트리 번들과 공통 청크의 책임을 나눠서 보기 쉬워진다.

### 고민한 대안

- 기본 설정 유지
- runtime 코드 분리를 별도로 고려하지 않음

### 결과 / 영향

런타임 코드가 별도 청크로 분리되어 번들 구조 해석이 쉬워졌고, production 번들 구성을 더 명확하게 이해할 수 있었다.

---

## 7. production에서 CSS를 별도 파일로 추출한 이유

### 결정

운영 빌드에서는 `MiniCssExtractPlugin`을 사용해 CSS를 별도 파일로 추출했다.

### 배경

개발 환경에서는 `style-loader`를 통해 스타일을 빠르게 주입하는 방식이 편리하지만, 운영 환경에서는 CSS가 JS 번들에 함께 포함되면 초기 JS 번들 크기에 부담을 줄 수 있다.

### 선택한 이유

운영 환경에서는 CSS를 별도 파일로 분리해 정적 리소스로 관리하는 편이 더 적절하다고 판단했다.  
이렇게 하면 스타일과 스크립트의 역할이 분리되고, 브라우저가 CSS 파일을 독립적으로 처리할 수 있다.

### 결과 / 영향

CSS 추출 적용 후에는 운영 빌드 결과물 기준으로 스타일 코드가 JS 번들 내부에 포함되는 대신 `dist`에 별도 CSS 파일로 생성되는 것을 확인했다.  
또한 초기 JS 번들 부담을 줄이는 방향으로 구조를 개선할 수 있었고, entrypoint size warning 완화에도 도움이 됐다.

---

## 8. 환경변수 공개 범위를 `APP_PUBLIC_` prefix로 제한한 이유

### 결정

클라이언트에 주입되는 환경변수는 `APP_PUBLIC_` prefix가 붙은 값으로 제한했다.

### 배경

환경변수를 무분별하게 브라우저 번들에 주입하면, 서버 전용 값이나 민감한 값이 의도치 않게 노출될 위험이 있다.

### 선택한 이유

브라우저에 노출 가능한 값만 명시적으로 공개하도록 규칙을 두면, 템플릿 단계에서부터 공개/비공개 기준을 일관되게 유지할 수 있다. Next.js의 `NEXT_PUBLIC_`처럼 prefix 기반 공개 정책은 이해하기도 쉽고 실수 방지에도 도움이 된다.

### 고민한 대안

- 모든 `.env` 값을 그대로 주입
- 별도 규칙 없이 개발자 판단에 의존

### 결과 / 영향

클라이언트 노출 가능 값과 서버 전용 값을 명확히 구분할 수 있게 되었고, 환경변수 관리 기준을 문서화할 수 있었다.

---

## 9. 환경변수 파일 구조를 .env 계열로 분리한 이유

### 결정

`.env`, `.env.local`, `.env.[mode]`, `.env.[mode].local` 구조를 지원하도록 했다.

### 배경

공통 설정, 개인 로컬 설정, 실행 모드별 설정을 하나의 파일에 모두 담으면 관리가 복잡해지고, 협업이나 재사용 가능한 템플릿 구조로 확장하기 어려워진다.

### 선택한 이유

환경에 따라 값을 나눌 수 있도록 구조를 먼저 잡아두면, 이후 템플릿 확장이나 실제 서비스 환경으로 발전시킬 때도 일관된 기준을 유지할 수 있다.

### 고민한 대안

- `.env` 하나만 사용
- 로컬/운영 설정 차이를 코드 내부에서 직접 분기

### 결과 / 영향

공통 설정, 로컬 전용 값, 실행 모드별 값을 분리해 관리할 수 있게 되었고, 템플릿 단계에서부터 환경 분리 기준을 명확히 할 수 있었다.

---

## 10. 번들 결과를 Analyzer와 Network 탭으로 함께 검증한 이유

### 결정

코드 스플리팅 결과는 `webpack-bundle-analyzer`와 브라우저 Network 탭으로 함께 확인했다.

### 배경

설정을 추가했다고 해서 실제로 원하는 형태로 번들이 분리되었다고 단정할 수는 없다.  
정적인 번들 구조와 실제 브라우저 요청 흐름은 함께 검증해야 한다.

### 선택한 이유

Bundle Analyzer는 어떤 코드가 어떤 청크에 포함되었는지 정적으로 보여주고, Network 탭은 실제 라우트 전환 시 어떤 파일이 언제 요청되는지 동적으로 보여준다. 둘을 함께 봐야 설정 결과를 더 정확하게 확인할 수 있다고 판단했다.

### 고민한 대안

- 설정 적용 여부만 코드 수준에서 확인
- Analyzer만 사용하거나 Network 탭만 단독 사용

### 결과 / 영향

초기 번들, 공통 청크, 라우트별 비동기 청크 분리 여부를 더 명확하게 확인할 수 있었고, “설정했다”가 아니라 “실제로 동작한다”까지 검증할 수 있었다.

---

## 정리

이 프로젝트에서 Webpack 설정은 단순히 앱을 실행하기 위한 boilerplate가 아니라, 개발 환경과 운영 환경의 역할을 분리하고, 라우트 단위 코드 스플리팅, 공통 청크 분리, SPA 라우팅 대응, 환경변수 공개 정책을 직접 설계하고 검증하기 위한 기준으로 사용했다. 따라서 이 문서는 설정값 자체보다도, 왜 이 옵션을 선택했고 어떤 문제를 해결하려 했는지를 기록하는 데 목적이 있다.
